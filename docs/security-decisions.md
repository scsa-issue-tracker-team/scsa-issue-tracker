# Security Decisions

이 문서는 SCSA Issue Tracker를 만들면서 보안 관점에서 어떤 위험을 인식했고, 현재 단계에서 어떤 결정을 했는지 정리한 기록이다.

목표는 보안을 과하게 포장하는 것이 아니라, 작은 프로젝트 안에서도 기본적인 보안 원칙을 의식하면서 구현하기 위함이다.

## 기본 방향

SCSA Issue Tracker는 사용자, 프로젝트, 이슈, 댓글을 다루는 협업용 시스템이다. 따라서 단순히 기능이 동작하는 것보다 "정해진 사용자에게만 안전하게 동작하는가"를 함께 고려한다.

현재 단계에서는 다음 기준을 우선한다.

- 비밀번호는 평문으로 저장하지 않는다.
- 로그인 실패 원인을 과하게 노출하지 않는다.
- 클라이언트가 보낸 사용자 식별자를 그대로 신뢰하지 않는다.
- 프로젝트, 이슈, 댓글은 소속과 권한을 확인한 뒤 접근하게 한다.
- 개발자에게 필요한 상세 원인은 서버 로그에 남기고, 사용자 응답은 필요한 수준으로 제한한다.

## 비밀번호 저장

회원가입 시 사용자의 비밀번호는 BCrypt로 해시한 뒤 저장한다.

평문 비밀번호를 DB에 저장하면 DB가 노출되었을 때 사용자 계정이 즉시 위험해진다. BCrypt는 같은 비밀번호라도 매번 다른 해시를 만들고, 계산 비용을 높여 무차별 대입 공격을 어렵게 만든다.

현재 구현에서는 `PasswordEncoder`를 통해 비밀번호를 저장하고 검증한다.

- 회원가입: 입력 비밀번호를 BCrypt 해시로 변환 후 저장
- 로그인: 입력 비밀번호와 DB의 해시 값을 비교
- 응답: 비밀번호는 절대 반환하지 않음

## 로그인 실패 응답

로그인 실패 시 아이디가 없는 경우와 비밀번호가 틀린 경우를 구분해서 알려주지 않는다.

두 경우를 다르게 응답하면 공격자가 "존재하는 아이디"를 추측하는 데 힌트를 얻을 수 있다. 그래서 현재 로그인 실패는 동일하게 처리한다.

```text
401 Unauthorized
아이디 또는 비밀번호가 올바르지 않습니다.
```

이 방식은 사용자에게 필요한 수준의 안내는 제공하면서, 공격자에게 계정 존재 여부를 직접 알려주지 않기 위한 결정이다.

## JWT 사용 결정

JWT는 "유행이라서"가 아니라, 프론트엔드와 백엔드가 분리된 REST API 구조에서 인증 정보를 요청마다 전달하기 위한 방법으로 사용한다.

현재 구현한 범위는 로그인 성공 시 Access Token을 발급하는 단계다.

```json
{
  "tokenType": "Bearer",
  "accessToken": "..."
}
```

JWT 안에는 최소한의 식별 정보만 넣는다.

- userId
- username
- issuedAt
- expiration

비밀번호, 주민번호, 상세 개인정보 같은 민감 정보는 JWT에 넣지 않는다. JWT의 payload는 암호화된 비밀 영역이 아니라 디코딩 가능한 데이터이기 때문이다.

현재 Access Token 유효 시간은 30분으로 설정했다. 실무 수준에서는 secret을 코드나 properties 파일에 직접 두지 않고 환경변수나 Secret Manager로 관리해야 한다.

## Bearer 토큰 방식

로그인 후 클라이언트는 다음 요청부터 HTTP Authorization 헤더에 토큰을 담아 보낸다.

```http
Authorization: Bearer <accessToken>
```

`Bearer`는 "이 토큰을 가진 사람"이라는 의미의 인증 방식이다. 따라서 토큰이 유출되면 그 시간 동안 사용자처럼 요청할 수 있으므로, 토큰은 로그에 찍거나 화면에 노출하지 않아야 한다.

## 에러 응답

공통 예외 처리를 두어 에러 응답 형식을 통일한다.

현재 응답에는 상태 코드, 에러 코드, 사용자 메시지, 요청 경로, timestamp를 포함한다. 다만 서버 내부 예외의 상세 원인, SQL, 클래스명, 메서드명, stack trace는 사용자 응답에 노출하지 않는다.

예상하지 못한 오류는 사용자에게 일반 메시지로 응답하고, 자세한 원인은 서버 로그에 남긴다.

```text
사용자 응답: 서버 내부 오류가 발생했습니다.
서버 로그: 실제 예외 stack trace 기록
```

이 결정은 정당한 사용자가 다음 행동을 이해할 정도의 정보는 제공하되, 공격자에게 내부 구조 힌트를 주지 않기 위함이다.

## 권한 검증과 IDOR 방지

프로젝트, 이슈, 댓글은 URL에 있는 id만 믿고 조회하면 안 된다. 예를 들어 사용자가 `projectId`나 `issueId`를 바꿔서 다른 사람의 데이터에 접근할 수 있기 때문이다.

이를 막기 위해 앞으로의 구현에서는 다음 원칙을 따른다.

- `GET /api/v1/projects`는 전체 프로젝트가 아니라 로그인한 사용자가 참여한 프로젝트만 반환한다.
- `GET /api/v1/projects/{projectId}`는 로그인한 사용자가 해당 프로젝트 멤버인지 확인한다.
- 이슈와 댓글은 해당 프로젝트에 속한 데이터인지 확인한다.
- 권한이 없으면 403 Forbidden으로 차단한다.

즉 URL 구조가 DB 조회 순서를 의미하는 것이 아니라, 서버 내부에서 로그인 사용자 기준으로 접근 가능 여부를 판단한다.

## 개발 단계에서 허용한 것

현재 프로젝트는 학습과 협업 연습을 위한 초기 단계이므로 일부 항목은 개발 편의를 위해 열어두었다.

- Swagger UI는 로컬 개발 중 API 확인을 위해 허용한다.
- JWT secret은 개발 편의를 위해 properties에 둔다.
- Refresh Token, 토큰 재발급, 로그아웃 토큰 무효화는 아직 구현하지 않았다.
- 요청 횟수 제한, 계정 잠금, 감사 로그는 아직 구현하지 않았다.

이 항목들은 운영 환경으로 간다면 반드시 다시 검토해야 한다.

## 다음 보안 작업

앞으로 이어서 구현할 보안 작업은 다음과 같다.

- JWT 검증 필터 추가
- Authorization 헤더에서 Bearer 토큰 읽기
- 토큰 검증 후 현재 사용자 식별
- 인증이 필요한 API 보호
- 프로젝트 멤버 여부 기반 인가 처리
- Swagger에서 Authorize 버튼으로 토큰 테스트
- JWT secret 환경변수 분리

## 정리

현재 보안 설계는 완성형이 아니라 시작점이다. 하지만 비밀번호 평문 저장을 피하고, 로그인 실패 응답을 일반화하고, JWT에 민감 정보를 넣지 않으며, 프로젝트 단위 권한 검증을 고려하는 방향으로 구현을 진행한다.

이 프로젝트의 핵심 기준은 다음과 같다.

```text
기능이 동작하는 것에서 끝내지 않고,
누가 어떤 데이터에 접근할 수 있는지까지 함께 설계한다.
```
