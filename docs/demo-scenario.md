# SCSA Issue Tracker 시연 시나리오

이 문서는 발표나 리뷰 상황에서 SCSA Issue Tracker를 어떤 순서로 보여줄지 정리한 시연 대본입니다.

## 시연 목표

단순히 화면이 있다는 것을 보여주는 것이 아니라, 다음 흐름을 보여주는 것이 목표입니다.

```text
사용자 인증
→ 프로젝트 생성
→ 멤버 초대
→ 이슈 등록
→ 이슈 상태 변경
→ 댓글로 협업
```

## 사전 준비

### 백엔드 실행

```bash
cd backend
./gradlew.bat bootRun
```

백엔드 주소:

```text
http://localhost:8081
```

Swagger UI:

```text
http://localhost:8081/swagger-ui/index.html
```

### 프론트엔드 실행

```bash
cd frontend
npm install
npm run dev
```

프론트 주소:

```text
http://localhost:5173
```

## 시연 순서

### 1. 회원가입

프론트 로그인 화면에서 회원가입 탭을 사용합니다.

예시:

```text
username: demo_owner
email: demo_owner@example.com
password: 1234
```

강조할 점:

- 비밀번호는 평문 저장이 아니라 BCrypt 해시로 저장합니다.
- 회원가입 응답에는 password를 포함하지 않습니다.

### 2. 로그인

방금 만든 계정으로 로그인합니다.

강조할 점:

- 로그인 성공 시 JWT access token을 발급합니다.
- 이후 API 요청은 `Authorization: Bearer <token>` 헤더로 인증합니다.
- 프론트는 토큰을 API client 계층에서 자동으로 붙입니다.

### 3. 프로젝트 생성

프로젝트 목록 화면에서 새 프로젝트를 생성합니다.

예시:

```text
name: SCSA 운영 개선
description: DS, DX, SDS 조직 공통 요청과 이슈를 관리하는 프로젝트
```

강조할 점:

- 프로젝트 생성자는 서버가 JWT에서 결정합니다.
- 프론트가 `createdById`를 보내지 않습니다.
- 프로젝트 생성자는 자동으로 `OWNER` 멤버가 됩니다.

### 4. 프로젝트 목록과 상세 진입

생성된 프로젝트가 목록에 보이는지 확인하고 상세 화면으로 진입합니다.

강조할 점:

- 프로젝트 목록은 현재 로그인 사용자가 참여한 프로젝트만 조회합니다.
- `GET /api/v1/projects`는 `userId` 쿼리 파라미터를 받지 않습니다.

### 5. 멤버 추가

다른 사용자 계정을 미리 생성해 두고 프로젝트 멤버로 추가합니다.

예시:

```text
username: demo_member
```

강조할 점:

- 멤버 추가는 `OWNER`만 가능합니다.
- 중복 멤버 추가는 `409 Conflict`로 막습니다.
- 프로젝트 소유자는 삭제할 수 없습니다.

### 6. 이슈 생성

프로젝트 상세 화면에서 이슈를 생성합니다.

예시:

```json
{
  "title": "월간 리포트 데이터 검증 요청",
  "content": "5월 매출 리포트의 일부 지표가 이전 데이터와 맞지 않아 확인이 필요합니다.",
  "issueType": "REQUEST",
  "priority": "MEDIUM",
  "assigneeId": 2
}
```

강조할 점:

- 이슈 작성자 `reporterId`는 JWT에서 서버가 결정합니다.
- 생성 시 `status`는 요청으로 받지 않고 서버가 `OPEN`으로 고정합니다.
- 담당자 `assigneeId`는 프로젝트 멤버여야 합니다.

### 7. 이슈 목록과 상세 조회

생성된 이슈가 목록에 표시되는지 확인하고 상세 화면으로 이동합니다.

강조할 점:

- 이슈 목록은 프로젝트 하위 자원입니다.
- 프로젝트 멤버가 아닌 사용자는 조회할 수 없습니다.
- 상태, 타입, 우선순위 필터를 지원합니다.

### 8. 이슈 상태 변경

이슈 상세 화면에서 상태를 변경합니다.

예시 흐름:

```text
OPEN → IN_PROGRESS → RESOLVED → CLOSED
```

강조할 점:

- 일반 이슈 수정 API와 상태 변경 API를 분리했습니다.
- 상태 변경은 전용 API를 사용합니다.

```text
PATCH /api/v1/projects/{projectId}/issues/{issueId}/status
```

### 9. 댓글 작성

이슈 상세 화면에서 댓글을 작성합니다.

예시:

```text
지표 원본 데이터와 비교해보겠습니다.
```

강조할 점:

- 댓글 작성자 `authorId`는 JWT에서 서버가 결정합니다.
- 댓글은 프로젝트 멤버만 작성하고 조회할 수 있습니다.

### 10. 권한 검증 확인

가능하다면 다른 사용자로 로그인해서 접근 권한을 확인합니다.

확인할 수 있는 상황:

- 프로젝트 멤버가 아닌 사용자가 프로젝트 상세 접근 시도
- 프로젝트 멤버가 아닌 사용자가 이슈 상세 접근 시도
- `OWNER`가 아닌 사용자가 멤버 추가 시도

기대 결과:

```text
403 Forbidden
```

## 발표 때 강조할 포인트

### 기능보다 중요한 설계 포인트

- 프론트가 사용자 ID를 임의로 보내지 않고, 서버가 JWT에서 현재 사용자를 판단합니다.
- 프로젝트 멤버 기반으로 접근 권한을 검증합니다.
- 이슈 담당자도 프로젝트 멤버인지 검증합니다.
- 이슈 상태 변경을 별도 API로 분리해 의도가 명확합니다.
- 웹뿐 아니라 Android에서도 같은 REST API와 JWT 계약을 사용할 수 있습니다.

### 보안 의식

- password를 응답하지 않습니다.
- password를 평문 저장하지 않습니다.
- 클라이언트가 보내는 `userId`, `reporterId`, `authorId`를 신뢰하지 않습니다.
- 권한 없는 사용자의 프로젝트/이슈/댓글 접근을 막습니다.
